üé≠ Field Masking Types

The system implements six distinct masking types, each designed for specific privacy and security requirements.

NONE No Masking

Complete data visibility with no restrictions applied. Used for users with full access privileges.

  • Shows original data values
  • No data transformation
  • Full field visibility
  • Used for Central Workers
Original: "john.doe@company.com"
Masked: "john.doe@company.com"

HIDDEN Complete Hide

Completely hides sensitive data by replacing it with a generic placeholder.

  • Replaces with "***HIDDEN***"
  • No data leakage
  • Maximum privacy protection
  • Used for highly sensitive data
Original: "john.doe@company.com"
Masked: "***HIDDEN***"

PARTIAL_MASK Partial Mask

Shows partial data while hiding sensitive portions using customizable patterns.

  • Customizable masking patterns
  • Shows partial information
  • Maintains data format
  • Used for SSNs, phone numbers
Original: "123-45-6789"
Masked: "XXX-XX-6789"

HASH_MASK Hash Mask

Replaces sensitive data with cryptographic hash values for anonymization while maintaining uniqueness.

  • Cryptographic hash generation
  • Maintains data uniqueness
  • One-way transformation
  • Used for IDs and identifiers
Original: "PROV_12345"
Masked: "HASH_1234567890"

ANONYMIZE Anonymize

Replaces personal information with generic, anonymized values while preserving data structure.

  • Generic value replacement
  • Preserves data structure
  • Removes personal identifiers
  • Used for names and personal data
Original: "John Doe"
Masked: "User 123"

AGGREGATE Aggregate

Shows data in aggregated ranges or categories instead of exact values for privacy protection.

  • Range-based aggregation
  • Category-based grouping
  • Statistical privacy
  • Used for financial amounts
Original: "$2,500.00"
Masked: "$1000-5000"

üîÑ Implementation Workflow

The field masking system follows a structured workflow to ensure proper data protection and access control.

Field Masking Implementation Process

1. User Request
2. Role Validation
3. Rule Retrieval
4. Data Processing
5. Masking Application
6. Response Delivery

Detailed Process Steps

  1. User Request: User submits request for data with specific role and context
  2. Role Validation: System validates user role and associated permissions
  3. Rule Retrieval: System retrieves field masking rules for the user role
  4. Data Processing: Raw data is processed and prepared for masking
  5. Masking Application: Appropriate masking rules are applied to each field
  6. Response Delivery: Masked data is returned to the requesting user

Masking Rule Application Logic

// Pseudo-code for masking rule application for (Field field : dataFields) { FieldMaskingRule rule = getMaskingRule(field.getName(), userRole); switch (rule.getMaskingType()) { case NONE: // No masking applied break; case HIDDEN: field.setValue("***HIDDEN***"); break; case PARTIAL_MASK: field.setValue(applyPartialMask(field.getValue(), rule.getMaskingPattern())); break; case HASH_MASK: field.setValue(generateHash(field.getValue())); break; case ANONYMIZE: field.setValue(generateAnonymizedValue(field.getName())); break; case AGGREGATE: field.setValue(applyAggregation(field.getValue(), field.getName())); break; } }
